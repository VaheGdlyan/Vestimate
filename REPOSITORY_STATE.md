# REPOSITORY STATE

## 1. Project Overview & Current State
**Vestimate** is an asynchronous backend system designed for an AI-powered wardrobe and styling application. It handles image ingestion, offloads heavy ML processing (background removal, item segmentation, and FashionCLIP vector embeddings) to serverless GPU endpoints, stores image assets in a CDN-backed object store, and persists metadata and high-dimensional vectors to a PostgreSQL database for semantic search and personalized styling recommendations.

**Current Phase Status**: Phase 2 is **COMPLETE**.
* ✅ The Python 3.11 virtual environment is stable and dependency conflicts are resolved.
* ✅ Docker & Docker Compose are configured and actively running services (e.g., Redis).
* ✅ Supabase (PostgreSQL with `pgvector`) is fully integrated and the database schema is deployed.
* ✅ Cloudflare R2 is configured and actively functioning as the object storage layer.

## 2. Technology Stack
* **Language:** Python 3.11
* **Framework:** FastAPI
* **Database:** Supabase (PostgreSQL) with `pgvector` for embeddings. (Uses direct `supabase-py` client for REST insertions, `asyncpg`/SQLAlchemy available for relational queries).
* **Storage:** Cloudflare R2 (S3-compatible, accessed via `boto3`)
* **Background Processing:** Celery with Redis broker
* **ML Inference:** Modal (Serverless GPU endpoints for Rembg and FashionCLIP)
* **Infrastructure:** Docker & Docker Compose (for local Redis/broker orchestration)

## 4. Core Components & Logic

### Database Models (Supabase / PostgreSQL)
The database schema relies heavily on UUIDs and foreign keys. Current tables include:
* `users`: Core user accounts (required for all foreign key references).
* `wardrobe_items`: Stores garment metadata (category, color, fit, material), R2 image keys, ML confidence scores, and a 512-dimension vector `embedding` column powered by `pgvector`.
* `outfits`: Records of assembled outfits (top, bottom, shoes) generated by the system or user.
* `recommendation_cache`: Stores generated outfit recommendations with weather/schedule context.
* `feedback_events`: Audit trail of user interactions (worn, skipped, saved) on recommendations.
* `prompt_versions`: Manages versioning for LLM styling prompt templates.
* `manual_review_queue`: Holds items that received a low ML confidence score (< 0.70) during ingestion.
* `event_log`: Append-only audit trail for system events.

### API Endpoints (FastAPI)
* `GET /health`: Basic health check ensuring the API is responsive.
* `POST /v1/wardrobe/upload`: Accepts an image file and `user_id`. It immediately uploads the raw image to Cloudflare R2, creates a "processing" stub record in Supabase, and enqueues a Celery task for asynchronous ML processing.
* `GET /v1/tasks/{task_id}`: Polling endpoint for clients to check the real-time status (PENDING, PROCESSING, SUCCESS, FAILED) of a background Celery task.

### Storage Logic (Cloudflare R2)
Storage is handled in `app/services/storage.py` using `boto3`. 
* The system connects to R2 using an S3-compatible API.
* Raw images are uploaded synchronously during the FastAPI request.
* Pre-signed URLs are generated by the Celery worker to securely pass the image to external Modal GPU endpoints for processing without exposing the R2 bucket publicly.
* Processed (segmented) images are uploaded back to R2 by the worker.

### Background Workers (Celery/Redis)
* Configured in `app/worker/celery_app.py` using Redis as the message broker.
* Contains the primary pipeline task: `ingest_garment` (located in `tasks.py`).
* **Workflow**: 
  1. Downloads the raw image from R2 via pre-signed URL.
  2. Sends the image to the Modal Segmentation endpoint (Rembg).
  3. Uploads the transparent segmented PNG back to R2.
  4. Sends the segmented image to the Modal Embedding endpoint (FashionCLIP) to extract semantic tags and the 512-d vector.
  5. Updates the Supabase `wardrobe_items` record with the results, routing low-confidence items to the `manual_review_queue`.
  6. Includes robust error handling, marking the database record as "failed" if the pipeline crashes.

## 5. Environment Configuration
The system relies on the following environment variables (defined in `.env.example`). The `app/core/config.py` file strictly validates their presence on startup:

* `APP_NAME`
* `DEBUG`
* `REDIS_URL`

**Supabase Configuration:**
* `SUPABASE_URL`
* `SUPABASE_SERVICE_KEY` (Requires the `service_role` JWT secret for backend operations)
* `SUPABASE_DATABASE_URL` (Direct connection string, `postgresql+asyncpg://` prefix required)

**Cloudflare R2 Configuration:**
* `R2_ACCOUNT_ID`
* `R2_ACCESS_KEY_ID`
* `R2_SECRET_ACCESS_KEY`
* `R2_BUCKET_NAME`

**Modal Inference Endpoints:**
* `MODAL_ENDPOINT_SEGMENT`
* `MODAL_ENDPOINT_EMBED`
